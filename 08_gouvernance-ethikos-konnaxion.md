# 08 — Gouvernance Ethikos / Konnaxion

**Projet :** Orphée: Walk Straight  
**Chemin :** `orphee-walk-straight/08_gouvernance-ethikos-konnaxion.md`  
**Statut :** Document de cadrage  
**Version :** 0.1  
**Fonction :** Définir le rôle d’Ethikos/Konnaxion dans la gouvernance collective du projet, sans confondre l’outil avec l’autorité humaine, pédagogique ou institutionnelle.

---

## 1. Résumé

Dans le Projet Orphée: Walk Straight, **Ethikos/Konnaxion** sert de couche de gouvernance collective.

Son rôle n’est pas de former directement les apprenants.  
Son rôle n’est pas de remplacer les enseignants, le cégep, les experts, les partenaires ou les assemblées humaines.  
Son rôle n’est pas de décider automatiquement.

Son rôle est de soutenir une communauté dans les opérations suivantes :

- formuler les enjeux;
- consulter les participants;
- prioriser les besoins;
- structurer les propositions;
- recueillir des arguments;
- documenter les décisions;
- suivre les actions;
- conserver la mémoire collective;
- rendre les choix contestables, vérifiables et améliorables.

Formule courte :

> **UCKK forme. Ethikos/Konnaxion gouverne la discussion. Les humains décident. Les archives prouvent.**

---

## 2. Principe directeur

Le Projet Orphée repose sur une distinction fondamentale :

```text
Formation ≠ Gouvernance
Gouvernance ≠ Décision automatique
Décision ≠ Vérité absolue
Archive ≠ Immobilité
````

La formation se déroule principalement dans les environnements pédagogiques du projet : UCKK, Moodle, ressources de cours, ateliers, documents, exercices et productions concrètes.

La gouvernance collective se déroule dans Ethikos/Konnaxion, ou dans des processus inspirés par Ethikos/Konnaxion lorsque le déploiement technique complet n’est pas encore requis.

Ethikos/Konnaxion doit aider la communauté à mieux choisir, non à choisir à sa place.

---

## 3. Rôle d’Ethikos/Konnaxion dans le projet

Ethikos/Konnaxion agit comme un outil de gouvernance pour :

1. **prioriser les contenus à produire;**
2. **identifier les besoins des apprenants et partenaires;**
3. **organiser les rétroactions;**
4. **documenter les arbitrages pédagogiques;**
5. **conserver les raisons derrière les décisions;**
6. **suivre l’amélioration continue;**
7. **préparer les transferts vers les milieux isolés ou contraints.**

Il peut être utilisé pendant :

* le pilote local hybride;
* les ateliers de production pédagogique;
* les cycles de révision de contenus;
* les consultations avec partenaires;
* les transferts vers la Basse-Côte-Nord;
* les adaptations futures pour milieux fermés ou carcéraux.

---

## 4. Ce que Konnaxion n’est pas

Pour éviter toute confusion institutionnelle, Konnaxion ne doit pas être présenté comme :

* une autorité scolaire;
* un ministère;
* une plateforme d’accréditation publique;
* un système disciplinaire;
* un système de surveillance;
* un outil de notation sociale;
* un mécanisme politique général;
* un système automatisé de décision;
* un substitut aux enseignants;
* un substitut au cégep;
* un substitut aux règles des partenaires.

Konnaxion est un outil de soutien à la gouvernance collective.
Il ne possède aucune autorité légitime par lui-même.

L’autorité vient des personnes, institutions et processus reconnus :

* le cégep;
* les enseignants;
* les partenaires;
* les comités;
* les assemblées;
* les règles du projet;
* les ententes officielles;
* les responsabilités légales et éthiques.

---

## 5. Place d’Ethikos dans Orphée: Walk Straight

Ethikos est le démonstrateur ou l’interface de gouvernance liée à Konnaxion.

Dans ce projet, Ethikos sert à rendre visible :

* les questions ouvertes;
* les propositions;
* les arguments;
* les priorités;
* les objections;
* les décisions;
* les traces;
* les suivis;
* les apprentissages collectifs.

Ethikos doit être présenté comme une **interface de délibération structurée**.

Il peut servir à répondre à des questions comme :

* Quel module de formation doit être produit en premier?
* Quels contenus sont les plus utiles pour une cohorte locale?
* Quelles compétences doivent être priorisées?
* Quels formats conviennent aux milieux à faible connectivité?
* Quels risques doivent être traités avant un transfert?
* Quels critères de qualité doivent être adoptés?
* Quelles productions sont prêtes à être diffusées?
* Quelles productions doivent être révisées?
* Quelles adaptations sont nécessaires pour la Basse-Côte-Nord?
* Quelles adaptations seraient nécessaires pour un milieu carcéral?

---

## 6. Principe : gouverner le programme, pas les personnes

Ethikos/Konnaxion ne doit jamais être utilisé pour gouverner les personnes de façon coercitive.

Le système doit gouverner :

* les contenus;
* les priorités;
* les décisions de projet;
* les processus de validation;
* les versions de documents;
* les critères de qualité;
* les demandes d’amélioration;
* les cycles de transfert.

Il ne doit pas servir à :

* classer moralement les participants;
* surveiller les comportements individuels;
* prédire la valeur d’une personne;
* imposer une décision automatisée;
* remplacer le jugement humain;
* exclure une personne sans processus humain clair.

Dans Orphée: Walk Straight, la gouvernance doit rester orientée vers la production de savoirs, la qualité pédagogique, l’inclusion et l’amélioration continue.

## 7. Gouvernance du pilote hybride local

Le premier pilote réunit une cohorte hybride locale :

* étudiants;
* membres du personnel;
* personnes de la communauté;
* experts invités;
* partenaires régionaux;
* contributeurs volontaires.

Ethikos/Konnaxion peut servir à organiser cette cohorte autour de cycles simples :

```text
Proposer → Discuter → Prioriser → Produire → Réviser → Valider → Archiver
```

### 7.1 Proposer

Les participants peuvent proposer :

* un module de cours;
* un sujet de capsule;
* un plan de livre;
* une ressource multimédia;
* une procédure;
* un glossaire;
* un outil pédagogique;
* une adaptation multilingue;
* une amélioration de parcours.

### 7.2 Discuter

Les propositions sont discutées selon des critères explicites :

* utilité pédagogique;
* faisabilité;
* accessibilité;
* valeur pour les publics visés;
* compatibilité avec les outils disponibles;
* effort requis;
* risques éthiques;
* potentiel de transfert.

### 7.3 Prioriser

Les priorités ne doivent pas être fondées seulement sur popularité ou enthousiasme.

Elles doivent tenir compte de :

* l’impact;
* l’urgence;
* la capacité réelle de production;
* les besoins des partenaires;
* les contraintes techniques;
* les contraintes linguistiques;
* les contraintes d’accessibilité;
* la valeur d’apprentissage.

### 7.4 Produire

Les contenus priorisés sont transformés en productions concrètes :

* documents;
* modules;
* exercices;
* quiz;
* présentations;
* guides;
* capsules;
* procédures;
* fiches;
* ressources exportables.

### 7.5 Réviser

Chaque production doit passer par une révision :

* pédagogique;
* technique;
* linguistique;
* éthique;
* accessibilité;
* cohérence documentaire;
* usage responsable de l’IA.

### 7.6 Valider

Une production n’est pas validée automatiquement par l’outil.

Elle est validée par les personnes ou instances désignées :

* enseignant;
* responsable pédagogique;
* comité de projet;
* expert invité;
* partenaire concerné;
* responsable institutionnel, si applicable.

### 7.7 Archiver

Les décisions et productions validées doivent être archivées avec :

* version;
* auteur ou contributeurs;
* date;
* statut;
* justification;
* critères utilisés;
* objections importantes;
* décisions de révision;
* liens vers les productions finales.

## 8. Gouvernance des terrains spécialisés

Après le pilote local, le projet vise des terrains spécialisés de transfert :

1. milieux isolés;
2. Basse-Côte-Nord;
3. villages éloignés;
4. environnements à faible connectivité;
5. milieux contraints;
6. éventuellement milieu carcéral.

Ethikos/Konnaxion doit aider à adapter le modèle sans imposer une solution uniforme.

Chaque terrain doit avoir son propre processus de consultation.

### 8.1 Milieux isolés

Pour les milieux isolés, la gouvernance doit examiner :

* connectivité réelle;
* disponibilité des équipements;
* langues utilisées;
* horaires possibles;
* accompagnement local;
* pertinence des contenus;
* accès hors ligne;
* modes de diffusion;
* soutien technique;
* besoins de formation continue.

### 8.2 Basse-Côte-Nord

Pour la Basse-Côte-Nord, la gouvernance doit intégrer :

* distances;
* isolement;
* réalités linguistiques;
* réalités communautaires;
* saisonnalité;
* transport;
* disponibilité des intervenants;
* infrastructures numériques;
* besoins des organismes locaux;
* respect des dynamiques territoriales.

### 8.3 Milieux contraints

Pour les milieux contraints, la gouvernance doit intégrer :

* sécurité;
* confidentialité;
* permissions;
* absence d’accès libre au web;
* supervision;
* consentement;
* règles institutionnelles;
* limites de production;
* gestion des données;
* acceptabilité sociale.

### 8.4 Milieu carcéral

Le milieu carcéral doit être abordé comme un terrain futur, spécialisé et sensible.

Toute expérimentation doit respecter les principes suivants :

* participation volontaire;
* absence de travail forcé;
* activité formative avant toute production utile;
* reconnaissance des apprentissages;
* absence de contact non supervisé avec l’extérieur;
* absence de données sensibles;
* absence d’activités pouvant favoriser des usages criminogènes;
* encadrement humain;
* validation institutionnelle;
* respect des programmes existants.

Ethikos/Konnaxion peut soutenir la gouvernance du programme, mais ne doit jamais devenir un outil de contrôle des personnes incarcérées.

## 9. Rôles de gouvernance

La gouvernance du projet doit distinguer les rôles suivants.

### 9.1 Porteur institutionnel

Le porteur institutionnel est l’organisation qui assume la responsabilité officielle du projet dans un cadre donné.

Dans une trajectoire CCTT, ce rôle pourrait être assumé par :

* le Cégep de Baie-Comeau;
* une unité de recherche;
* une équipe de formation continue;
* un laboratoire pré-CCTT;
* une structure partenariale reconnue.

### 9.2 Responsable pédagogique

Le responsable pédagogique veille à :

* la cohérence des parcours;
* la qualité des contenus;
* l’alignement avec les objectifs;
* la validité des exercices;
* la progression des compétences;
* l’évaluation;
* la reconnaissance des acquis, si applicable.

### 9.3 Responsable de gouvernance

Le responsable de gouvernance veille à :

* l’organisation des consultations;
* la clarté des décisions;
* la documentation des raisons;
* la gestion des conflits;
* l’archivage;
* la traçabilité;
* la cohérence entre décisions et actions.

### 9.4 Experts invités

Les experts invités peuvent contribuer à :

* valider des contenus;
* proposer des améliorations;
* évaluer la faisabilité;
* identifier des risques;
* aider au transfert vers un terrain spécialisé.

Ils ne doivent pas devenir une autorité opaque. Leur contribution doit être visible et contextualisée.

### 9.5 Participants

Les participants peuvent être :

* apprenants;
* contributeurs;
* testeurs;
* réviseurs;
* producteurs de contenus;
* membres d’un atelier;
* personnes consultées;
* usagers finaux.

Leur rôle doit toujours être clair.

### 9.6 Partenaires

Les partenaires peuvent inclure :

* organismes communautaires;
* municipalités;
* MRC;
* établissements d’enseignement;
* experts technopédagogiques;
* organismes de développement régional;
* milieux de formation continue;
* milieux spécialisés.

Un partenaire peut exprimer un besoin, mais ne doit pas imposer seul l’orientation pédagogique du projet.

## 10. Cycle de décision

Le cycle de décision recommandé est le suivant :

```text
Signal → Question → Proposition → Délibération → Lecture → Décision humaine → Action → Archive → Révision
```

### 10.1 Signal

Un signal est une information qui indique un besoin, un problème ou une opportunité.

Exemples :

* un apprenant ne comprend pas un module;
* un village manque d’accès à une formation;
* un partenaire demande une ressource;
* une procédure est trop complexe;
* une traduction est insuffisante;
* un outil n’est pas utilisable hors ligne.

### 10.2 Question

Le signal est transformé en question gouvernable.

Exemples :

* Quel module doit être simplifié?
* Quel format offline est prioritaire?
* Quelle compétence doit être enseignée avant la production multimédia?
* Quel contenu doit être adapté pour un public à faible connectivité?

### 10.3 Proposition

Une ou plusieurs propositions sont formulées.

Une bonne proposition doit contenir :

* objectif;
* public visé;
* production attendue;
* effort estimé;
* risques;
* critères de succès;
* responsable proposé;
* échéance possible.

### 10.4 Délibération

La délibération organise les arguments.

Elle doit distinguer :

* faits;
* hypothèses;
* préférences;
* contraintes;
* risques;
* objections;
* alternatives;
* conditions de réussite.

### 10.5 Lecture

Ethikos/Konnaxion peut produire une lecture structurée de la situation.

Cette lecture peut aider à voir :

* niveaux d’accord;
* objections récurrentes;
* priorités émergentes;
* tensions;
* risques;
* points nécessitant arbitrage.

La lecture n’est pas une décision.

### 10.6 Décision humaine

La décision appartient aux personnes ou instances responsables.

La décision doit indiquer :

* ce qui est décidé;
* pourquoi;
* qui est responsable;
* ce qui est exclu;
* quelles conditions doivent être respectées;
* comment la décision sera révisée.

### 10.7 Action

La décision est transformée en tâches.

Chaque tâche doit avoir :

* responsable;
* production attendue;
* échéance;
* critères d’acceptation;
* dépendances;
* statut.

### 10.8 Archive

L’archive conserve :

* la proposition;
* les arguments;
* la décision;
* les tâches;
* les versions;
* les livrables;
* les objections;
* les résultats.

### 10.9 Révision

Une décision peut être révisée lorsque :

* le contexte change;
* une erreur est identifiée;
* un partenaire signale un problème;
* les apprenants échouent à utiliser la ressource;
* un risque éthique apparaît;
* une meilleure option devient disponible.

---

## 11. Critères de priorité

Les propositions dans Ethikos/Konnaxion peuvent être priorisées selon les critères suivants.

| Critère         | Question                                                     |
| --------------- | ------------------------------------------------------------ |
| Utilité         | Est-ce que cela répond à un besoin réel?                     |
| Accessibilité   | Est-ce utilisable par des publics éloignés ou empêchés?      |
| Faisabilité     | Peut-on le produire avec les ressources disponibles?         |
| Sobriété        | Est-ce compatible avec des technologies légères et robustes? |
| Multilinguisme  | Peut-on l’adapter à plusieurs langues?                       |
| Offline         | Peut-il fonctionner hors ligne ou avec faible connectivité?  |
| Transférabilité | Peut-il servir dans plusieurs contextes?                     |
| Sécurité        | Est-ce que les risques sont maîtrisables?                    |
| Éthique         | Est-ce volontaire, non coercitif et respectueux?             |
| Pédagogie       | Est-ce que cela développe une compétence réelle?             |
| Traçabilité     | Peut-on documenter le processus et les décisions?            |

## 12. Garde-fous éthiques

Ethikos/Konnaxion doit intégrer des garde-fous explicites.

### 12.1 Consentement

Toute participation doit être volontaire, informée et révocable selon les règles du contexte.

### 12.2 Non-coercition

Aucune personne ne doit être forcée de produire, voter, participer, s’exposer ou contribuer.

### 12.3 Transparence des règles

Les règles de gouvernance doivent être connues avant la participation.

### 12.4 Droit à l’objection

Une objection raisonnable doit pouvoir être formulée, conservée et réexaminée.

### 12.5 Données minimales

Le projet doit collecter le minimum de données nécessaires.

### 12.6 Pas de notation sociale

Aucun score ne doit être utilisé pour classer moralement les personnes.

### 12.7 Décision humaine

Les décisions importantes doivent rester humaines et explicables.

### 12.8 Protection des publics vulnérables

Les publics vulnérables ou dépendants d’une institution doivent bénéficier de protections renforcées.

### 12.9 Sécurité des milieux contraints

En milieux fermés ou carcéraux, les règles de sécurité institutionnelles priment toujours.

### 12.10 Réversibilité

Une décision de gouvernance doit pouvoir être révisée si elle produit des effets indésirables.

## 13. Données, archives et confidentialité

La gouvernance collective exige des traces.
Mais la traçabilité ne doit pas devenir une surveillance.

Le projet doit distinguer :

* archives de décision;
* archives de production;
* archives pédagogiques;
* données personnelles;
* données sensibles;
* données anonymisées;
* données exportables;
* données internes.

### 13.1 Archives publiques

Certaines archives peuvent être publiques :

* décisions générales;
* critères de qualité;
* versions de documents;
* feuilles de route;
* contenus pédagogiques validés;
* rapports publics.

### 13.2 Archives internes

Certaines archives doivent rester internes :

* débats préparatoires;
* informations sur participants;
* commentaires non destinés à publication;
* évaluations;
* documents de travail;
* risques identifiés.

### 13.3 Archives restreintes

Certaines archives doivent être restreintes :

* informations sensibles;
* données personnelles;
* données liées à un milieu contraint;
* documents institutionnels;
* informations de sécurité.

### 13.4 Principe de minimisation

Le projet doit éviter de conserver inutilement des données personnelles.

Il faut archiver les décisions et les productions, pas exposer les personnes.

## 14. Usage de l’IA dans la gouvernance

L’IA peut aider à :

* résumer des discussions;
* comparer des propositions;
* détecter des doublons;
* reformuler des idées;
* produire des comptes rendus;
* identifier des critères;
* préparer des tableaux;
* générer des brouillons;
* traduire ou adapter du contenu;
* soutenir la documentation.

L’IA ne doit pas :

* décider;
* exclure;
* noter moralement;
* remplacer la délibération;
* effacer les objections;
* inventer des consensus;
* masquer l’incertitude;
* produire des comptes rendus sans validation humaine;
* traiter des données sensibles sans cadre clair.

Toute production IA utilisée dans la gouvernance doit être :

* signalée;
* vérifiée;
* révisée;
* validée;
* archivée avec mention de son rôle.

## 15. Gouvernance de la production pédagogique

Ethikos/Konnaxion peut structurer la production pédagogique selon ce cycle :

```text
Besoin → Plan → Prototype → Test → Révision → Validation → Diffusion → Retour d’usage
```

### 15.1 Besoin

Le besoin peut provenir :

* d’un apprenant;
* d’un enseignant;
* d’un partenaire;
* d’une communauté;
* d’un terrain spécialisé;
* d’un manque observé.

### 15.2 Plan

Le plan précise :

* objectif;
* public;
* niveau;
* compétences;
* format;
* outils;
* durée;
* livrables;
* critères de réussite.

### 15.3 Prototype

Le prototype peut être :

* un module;
* une présentation;
* une capsule;
* une fiche;
* un quiz;
* un guide;
* un glossaire;
* une procédure;
* un mini-livre;
* un paquet offline.

### 15.4 Test

Le test doit observer :

* compréhension;
* accessibilité;
* charge cognitive;
* faisabilité technique;
* valeur pratique;
* compatibilité offline;
* clarté des consignes.

### 15.5 Révision

La révision doit intégrer :

* rétroactions des apprenants;
* rétroactions des experts;
* contraintes techniques;
* erreurs détectées;
* besoins multilingues;
* adaptations low-tech.

### 15.6 Validation

La validation est humaine et institutionnelle lorsque nécessaire.

### 15.7 Diffusion

La diffusion peut se faire via :

* UCKK;
* Moodle;
* fichiers PDF;
* HTML statique;
* paquets téléchargeables;
* clé USB;
* intranet;
* médiathèque locale;
* partenaires.

### 15.8 Retour d’usage

Chaque contenu diffusé doit pouvoir générer des retours :

* ce qui fonctionne;
* ce qui bloque;
* ce qui doit être simplifié;
* ce qui doit être traduit;
* ce qui doit être adapté;
* ce qui doit être retiré.

## 16. Smart Vote et lectures collectives

Si Smart Vote ou un mécanisme apparenté est utilisé, il doit être compris comme une lecture collective, non comme une décision finale.

Smart Vote peut aider à voir :

* préférences;
* priorités;
* convergences;
* tensions;
* niveaux de confiance;
* objections;
* signaux faibles.

Mais Smart Vote ne doit pas :

* décider automatiquement;
* masquer les minorités;
* effacer les objections;
* transformer la popularité en vérité;
* servir de mécanisme disciplinaire;
* produire une autorité sans procédure.

Toute lecture Smart Vote doit être accompagnée de :

* contexte;
* critères;
* limites;
* trace;
* possibilité de contestation;
* validation humaine.

---

## 17. Contestation et révision

Une gouvernance saine doit inclure la contestation.

Une décision peut être contestée si :

* une règle n’a pas été respectée;
* une information importante a été oubliée;
* un groupe concerné n’a pas été consulté;
* une production crée un risque non prévu;
* une minorité importante est ignorée;
* une contrainte de terrain rend la décision impraticable;
* une erreur factuelle est détectée.

La contestation doit être :

* formulée clairement;
* reliée à une décision ou production;
* archivée;
* examinée;
* résolue ou maintenue comme objection active.

La contestation n’est pas un blocage.
Elle est un mécanisme d’intégrité.

## 19. Gouvernance minimale pour le pilote

Pour le premier pilote, ne pas surcomplexifier.

La gouvernance minimale doit inclure :

1. une liste de participants;
2. des rôles simples;
3. un tableau des propositions;
4. un processus de priorisation;
5. un registre des décisions;
6. un registre des productions;
7. un mécanisme de rétroaction;
8. une méthode de validation;
9. un dossier d’archives;
10. un rapport final.

Ethikos/Konnaxion peut soutenir ce processus, mais le pilote doit rester possible même avec des outils simples.

Principe :

> **La gouvernance doit être assez structurée pour apprendre, mais assez légère pour commencer.**

* rapports de révision;

* registre des contenus validés;
* rapport d’amélioration continue;
* documentation du modèle transférable.

## 21. Risques propres à la gouvernance

| Risque                       | Prévention                                       |
| ---------------------------- | ------------------------------------------------ |
| Outil trop central           | rappeler que les humains décident                |
| Participation symbolique     | relier les consultations à des décisions réelles |
| Surdocumentation             | limiter les archives aux traces utiles           |
| Biais de popularité          | utiliser plusieurs critères de priorité          |
| Confusion avec surveillance  | minimiser les données personnelles               |
| Confusion avec accréditation | séparer gouvernance et reconnaissance officielle |
| Conflit entre partenaires    | définir droits de décision avant le pilote       |
| Dépendance technique         | garder des formats exportables et offline        |
| IA trop influente            | exiger validation humaine                        |
| Carcéral trop sensible       | traiter comme terrain futur spécialisé           |

## 22. Règle d’or

La gouvernance d’Orphée: Walk Straight doit toujours respecter cette règle :

> **Un outil peut éclairer une décision, mais il ne doit jamais remplacer la responsabilité humaine de décider.**

Ethikos/Konnaxion est utile s’il augmente :

* la clarté;
* la mémoire;
* la participation;
* la qualité;
* la responsabilité;
* la capacité de transfert.

Il devient problématique s’il augmente :

* la dépendance;
* l’opacité;
* la surveillance;
* la coercition;
* la confusion institutionnelle;
* la délégation aveugle à la machine.
